<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client to API | X-API-Key / Bearer JWT / SSE token query-param cross an untrusted boundary |
| login to token | username + password posted to /api/auth/token |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Spoofing | api-key validation | mitigate | SHA-256 hash compare via constant-time MessageDigest.isEqual; never plaintext (AUTH-02) |
| T-03-05 | Spoofing | JWT verification | mitigate | java-jwt explicit Algorithm (HMAC256 dev / RS256 prod); never accept alg=none; verify exp/iat |
| T-03-06 | Elevation of privilege | authenticatedApi wrapper | mitigate | deny-by-default on protected routes when auth configured; fail-open ONLY when fully unconfigured (D-04a), proven by pass-through test |
| T-03-07 | Information disclosure | /api/auth/token | mitigate | uniform "Invalid credentials" for unknown-user AND bad-password; Argon2id constant-time verify; never log credentials |
| T-03-08 | Information disclosure | SSE token-in-query-param | accept | short-lived JWT; avoid logging full stream URLs; documented tradeoff (EventSource cannot set headers, Pitfall 2) |
| T-03-SC | Tampering | ktor-server-auth-jwt + password4j installs | mitigate | both OK in RESEARCH Package Legitimacy Audit (Maven Central, mature, JetBrains/Password4j); no [ASSUMED]/[SUS] |
</threat_model>
