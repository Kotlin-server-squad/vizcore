Output: `lib/auth-store.ts`, an extended `api-client.ts` (Bearer + 401 interception + token-aware SSE), a `/login` route per the UI-SPEC, and Vitest+MSW tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser to /api | JWT attached client-side; server is the enforcement point (client never validates) |
| localStorage | JWT at rest in browser storage |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-18 | Information disclosure | JWT in localStorage | accept | documented XSS-vs-UX tradeoff (D-06, ASVS V3); short-lived JWT; refresh later; no token logged |
| T-03-19 | Spoofing | client-side auth attach | mitigate | client only ATTACHES the credential; all validation is server-side (Plan 02) — no trust placed in the client |
| T-03-20 | Information disclosure | credentials in login form | mitigate | password field is type=password; creds never logged; posted only to the token endpoint over the same origin |
| T-03-21 | Information disclosure | 401 error copy | mitigate | generic "Incorrect username or password." (no user enumeration) — matches backend uniform 401 |
</threat_model>
