## Warnings

### WR-01: ConnectWizard mints an orphan backend session on every open and never cleans it up

**File:** `frontend/src/components/connect/ConnectWizard.tsx:48-68`
**Issue:** On every `isOpen` transition the effect calls
`createSession.mutateAsync(undefined)`, creating a brand-new backend session. If
the polled `coroutineCount` never transitions `> 0` (the user never wires up
their app — the *expected* path for someone just exploring the wizard) and they
click **Cancel** (or close the modal), the minted session is silently abandoned:
there is no `useDeleteSession`/cleanup call anywhere in the component (verified —
`delete` appears nowhere in the file). Each open leaks one empty session that
then shows up forever in the `SessionsSidebar` "Live apps" group (it has no
`scenario-` prefix → classified LIVE) with `0 active`. Repeated open/cancel
cycles accumulate orphan rows.
**Fix:** Track whether the minted session was actually adopted (navigated to) and
delete it on close otherwise. Minimal sketch:
```tsx
const deleteSession = useDeleteSession()
// in the open-effect cleanup, when not resolved:
return () => {
  cancelled = true
  if (!resolvedRef.current && newSessionId) {
    deleteSession.mutate(newSessionId) // discard the unused empty session
  }
}
```
(Alternatively, defer minting until the user actually commits to connecting, or
have the backend reap zero-event sessions after a TTL.)

### WR-02: Wizard poll query key collides with `useSession` and rewrites its polling cadence

**File:** `frontend/src/components/connect/ConnectWizard.tsx:73-78`
**Issue:** The wizard polls `useQuery({ queryKey: ['sessions', newSessionId],
refetchInterval: 300 })`. That key is byte-identical to `useSession`'s key
(`hooks/use-sessions.ts:13`). React Query merges observers on a shared key, so
while the wizard is mounted alongside anything observing `useSession(newSessionId)`
for the same id, the 300ms `refetchInterval` becomes the effective cadence of the
*shared* query (the global default is no interval; `useSession` sets none). In the
current flow this is benign because navigation unmounts the wizard, but it is a
silent coupling: any future code that renders the destination session while the
wizard is still open would inherit aggressive 300ms polling, and the wizard's
`getSession` snapshot pre-seeds/overwrites the route's cache entry. The data shapes
happen to match (`getSession → SessionSnapshot`, both expose `coroutineCount`), so
nothing breaks today — but the collision is undocumented and fragile.
**Fix:** Scope the wizard's poll to a distinct key so it can never merge with the
canonical session query:
```tsx
queryKey: ['connect-wizard-poll', newSessionId],
```
This keeps the wizard's transient 300ms poll isolated from `useSession`'s cache
entry and its own polling contract.

## Info

### IN-01: Auto-resolve transition is not announced to assistive tech

**File:** `frontend/src/components/connect/ConnectWizard.tsx:115-122`
**Issue:** Step 3's `Spinner` + "Waiting for events from {appName}…" silently
flips to a route navigation when events arrive. A screen-reader user gets no
announcement that the wait resolved — the region is not a live region and there
is no status role.
**Fix:** Wrap the waiting line in an `aria-live="polite"` status region, e.g.
`<div role="status" aria-live="polite">…Waiting for events…</div>`, so the
transition (and the eventual navigation) is perceivable.

### IN-02: Session-mint failure is swallowed with no surfaced error state

**File:** `frontend/src/components/connect/ConnectWizard.tsx:60-62`
**Issue:** The `.catch()` swallows a `createSession` rejection with the comment
"surfaced by the mutation state; the user can still Skip/Cancel" — but the
mutation state (`createSession.isError` / `.error`) is never read or rendered
anywhere in the component, and `Skip to live view` is `isDisabled={!newSessionId}`
(line 125), so on a mint failure `newSessionId` stays `undefined`, Skip is
disabled, and the user is left staring at a perpetual "Waiting for events…"
spinner with no error and no escape except Cancel. The comment overstates the UX.
**Fix:** Render the failure (e.g. `createSession.isError && <p className="text-sm
text-danger">Couldn't create a session — retry or run a demo instead.</p>`) and/or
keep an always-enabled escape affordance so a mint failure is not a silent dead end.

### IN-03: `deriveSessionKind` mislabels a user app whose name begins with `scenario-`

**File:** `frontend/src/lib/session-kind.ts:16`
**Issue:** The LIVE/DEMO split is a pure string-prefix heuristic
(`startsWith('scenario-')`). A genuine live app that happens to register with a
`scenario-`-prefixed session name would be badged DEMO. This is explicitly
documented as an accepted best-effort (PD-10 / T-085-08) and the live-by-default
fallback is the safe direction, so this is informational, not a defect — but it
is worth tracking until a backend discriminator field replaces the heuristic.
**Fix:** None required for this phase. When a backend `kind`/`source` field lands,
switch `deriveSessionKind` to read it and treat the prefix only as a fallback.
