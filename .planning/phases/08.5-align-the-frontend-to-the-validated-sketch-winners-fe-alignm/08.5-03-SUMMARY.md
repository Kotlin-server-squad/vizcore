# Phase 08.5 Plan 03: Connect / Onboarding Surface (Surface 003) Summary

Greenfield connect/onboarding surface — a badged sessions-sidebar-as-home with an unmistakable LIVE/DEMO pill on every row (derived client-side, no backend field) plus a 3-step HeroUI Modal connect wizard whose waiting Spinner auto-resolves to the live view via a new-id-scoped polled query.

## What Was Built

**Task 1 — badged sidebar + client-side LIVE/DEMO derivation (commit `12aa5ca`):**
- `lib/session-kind.ts` — pure `deriveSessionKind(session): 'live' | 'demo'`. A session is DEMO iff its `sessionId` carries the `scenario-` prefix that the scenarios route mints (`createSession.mutateAsync(`scenario-${name}`)`); everything else — including unknown/ambiguous — is LIVE (PD-10: the live app is the primary product, so a live session must never masquerade as a demo).
- `components/sessions/SessionRow.tsx` — one badged row: the reused `LivePill` (LIVE + "~150ms poll") for live, a neutral `DEMO` Chip for demo; label + mono `{N} active` meta from `coroutineCount` (the sketch's `{host}` is not wire-present in `SessionInfo`, so it degrades gracefully — Pitfall 3). Selected highlight `bg-primary/5 ring-2 ring-primary`, hover `bg-default-100` — literal-class branches, no `cn()` (IN-12).
- `components/sessions/SessionsSidebar.tsx` — a `320px` Card: header "Sessions" + primary `+ Connect`; the `useSessions()` list partitioned by `deriveSessionKind` into a "Live apps" group then a "Demo scenarios" group (each under a `text-xs uppercase tracking-wide text-default-400` header, omitted when empty); when the whole list is empty the "No app connected" empty state folds INLINE into the list with both CTAs ("Connect your app" → onConnect, "Run a demo scenario instead" → `/scenarios`) — not a standalone screen (UI-SPEC 145).

**Task 2 — 3-step ConnectWizard + route repoint (commit `e6dd4ae`):**
- `components/connect/ConnectWizard.tsx` — a HeroUI `Modal` titled "Connect your app" with three numbered step rows: (1) "Add the client library" + a copyable `Snippet` (`implementation("com.jh:vizcore-client:0.1")`), (2) "Enable in your app" + a copyable `Snippet` (`VizcoreClient.start(appName = "order-service")`) + sub-note "installs DebugProbes, streams to localhost:8080", (3) "Run your app" + a `Spinner` + mono "Waiting for events from order-service…". Footer: primary "Copy setup snippet" + ghost "Cancel"; a "Skip to live view" affordance navigates immediately as a fallback. On open it mints one session via `useCreateSession`, captures the new id, and POLLS that id via a LOCAL `useQuery({ queryKey: ['sessions', newId], enabled: !!newId, refetchInterval: 300 })` — when the polled `coroutineCount` transitions > 0 it AUTO-navigates to `/sessions/$sessionId` with no manual click (PD-11). Resolution is one-shot (`resolvedRef`) and scoped to the new id so it never fires off a previous session's cache (Pitfall 6).
- `routes/sessions/index.tsx` — repointed to render `<SessionsSidebar onConnect={openWizard} />` under `Layout` and mount `<ConnectWizard isOpen onClose />`; the old flat-grid `confirm()`-based create UI is gone (PD-12 — the create affordance is now `+ Connect`).

## Key Decisions

- **PD-11 poll mechanism — LOCAL query, not a useSession extension.** The shipped `useSession` has no `refetchInterval`, so it fetches the new session once at `coroutineCount: 0` and never transitions on its own. I chose a local new-id-scoped `useQuery` with `refetchInterval: 300` over threading an additive options param through `useSession`. Rationale: it keeps the change confined to the wizard and leaves every existing `useSession` call site byte-equivalent (zero regression surface).
- **`appName` is a static UI-SPEC constant** (`order-service`), rendered as plain React text (auto-escaped) and never injected into a URL/clipboard as markup (PD-14 / T-085-06). The snippets are static public copy — no token/secret embedded (PD-14 / T-085-07).

- **Fix:** Widened the param to the real call-site type `SessionInfo` (the function only reads `sessionId`, so behavior is unchanged).
- **Files modified:** `frontend/src/lib/session-kind.ts`
- **Commit:** `e6dd4ae`

- **Fix:** Replaced the unrecognized directives with plain explanatory comments documenting the intentional effect-dependency keying. No suppression needed — the rule isn't active.
- **Files modified:** `frontend/src/components/connect/ConnectWizard.tsx`
- **Commit:** `e6dd4ae`
