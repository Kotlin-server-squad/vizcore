<objective>
Close the timing-units gap: the backend `checksystem/TimingAnalyzer` (the one used by ValidationRoutes) computes durations from `tsNanos` and returns them in NANOSECONDS, while the frontend type `BackendTimingReport` documents the values as MILLISECONDS and `TimingReportView.formatMs()` treats them as ms. Result: a ~5s scenario renders ~109,172s total duration.

Decision (per UAT gap "Validation Timing Report" `missing`): make the backend response the source of truth in MILLISECONDS, matching the already-documented frontend contract, and add a magnitude-sanity test. This is the lower-blast-radius boundary: the FE type and formatMs are already correct for ms, so only the backend conversion is wrong.

Purpose: Run Validation shows believable durations end-to-end with no frontend change.
Output: ns→ms conversion in the backend TimingAnalyzer used by the validation route, updated KDoc unit documentation, and a test asserting ms-scale magnitudes.
