Delete the backend wrappers fork (reconciling into core first) and extend ForkDeletionTest to guard wrappers/ the same way it guards session/.

Purpose: FND-01's intent is "the running backend uses the single authoritative coroutine-viz-core classes." FND-03's intent is "a regression test asserts the fork cannot silently return." Both currently cover session/ only; this plan extends them to wrappers/.
Output: 11 deleted fork files; an extended ForkDeletionTest; a green test suite proving parity.

NOTE ON RECONCILIATION (verified during planning): a file-by-file diff of all 11 fork files against their core counterparts shows the ONLY differences are import style — the fork uses wildcard imports (`import com.jh.proj.coroutineviz.events.*`) while core uses explicit imports (ktlint no-wildcard-imports). There are ZERO non-import differences in any file (InstrumentedFlow, InstrumentedChannel, InstrumentedSharedFlow, InstrumentedStateFlow, VizMutex, VizSemaphore all show 0 non-import diff lines; the rest are byte-identical). VizScope.kt is byte-identical (the FIX-03 fix was applied to both copies). Therefore core is already the more-correct copy and NO behavioral content needs porting into core. The executor MUST re-confirm this with the diff command in Task 1 before deleting; if any non-import difference is found, it must port the newer/correct behavior into core first per decision D-02.
